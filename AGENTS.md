# Bronly Agent Rules

## Required reading order

Перед выполнением любой задачи агент обязан прочитать:

1. `README.md`
2. `docs/product/mvp-boundaries.md`
3. `docs/product/terminology.md`
4. `docs/process/task-intake.md`
5. `docs/product/prd.md` — если задача затрагивает продукт, UX, роли, публичные страницы, заявки, цены, подписку или агентскую модель
6. `docs/product/decision-log.md` — если задача меняет зафиксированную логику или уточняет спорные места

## Non-negotiable behavior

- Считать `docs/product/prd.md` главным продуктовым источником истины.
- Не придумывать новые продуктовые решения, если они не зафиксированы в документации.
- Во всех UI-текстах использовать терминологию проекта.
- Не добавлять out-of-scope функциональность без явного подтверждения пользователя.

## Conflict protocol

Если задача конфликтует с MVP, терминологией или зафиксированной бизнес-логикой, агент должен остановиться и:

1. назвать конфликт;
2. сослаться на конкретный документ и раздел;
3. предложить 1-2 безопасных варианта;
4. дождаться решения пользователя.

## Explicitly out of scope without approval

Без отдельного подтверждения пользователя не реализовывать:

- общий каталог объектов;
- глобальный поиск по всем владельцам;
- рекомендации похожих объектов;
- маркетплейс-логику;
- оплату проживания;
- автоматическое подтверждение заявки;
- гарантию проживания;
- возвраты денег гостю;
- интеграции с площадками бронирования;
- SMS-уведомления;
- сложный CRM-модуль;
- роли персонала владельца;
- автоматические расчеты между владельцем и агентом;
- публичный каталог агентов;
- мультиязычность на первом этапе.

## Feature intake rule

Каждая feature-задача должна ссылаться хотя бы на один раздел `docs/product/prd.md` или `docs/product/decision-log.md`. Если ссылки нет, агент обязан считать задачу неоднозначной и сначала уточнить продуктовое решение.
