# Bronly

Проект использует документацию как обязательный источник истины для продуктовых и UX-решений.

## Где лежит истина

- `docs/product/prd.md` — основной продуктовый документ, canonical PRD, `source of truth`.
- `docs/product/decision-log.md` — зафиксированные решения и будущие продуктовые уточнения.
- `docs/product/mvp-boundaries.md` — краткие границы MVP и признаки scope creep.
- `docs/product/terminology.md` — словарь UI/UX терминов и запрещенных формулировок.
- `docs/process/task-intake.md` — чеклист для любой новой задачи.
- `docs/process/ai-acceptance-prompts.md` — контрольные промпты для быстрой проверки поведения ИИ.

## Как ИИ должен работать в этом проекте

Перед началом задачи ИИ должен прочитать:

1. этот `README.md`;
2. `docs/product/mvp-boundaries.md`;
3. `docs/product/terminology.md`;
4. при продуктовых или UX-изменениях — `docs/product/prd.md`;
5. при спорных или новых решениях — `docs/product/decision-log.md`.

Все текстовые файлы в репозитории должны сохраняться в `UTF-8 without BOM`. Для проверки кодировки и типичных следов mojibake используйте `npm run check:encoding`.

Если задача конфликтует с MVP, терминологией или зафиксированной бизнес-логикой, ИИ не должен молча реализовывать изменение. Вместо этого он обязан:

1. назвать конфликт;
2. сослаться на конкретный документ или раздел;
3. предложить 1-2 безопасных варианта;
4. дождаться решения пользователя.

## Приоритет документов

1. `docs/product/decision-log.md` — если там явно зафиксировано более новое решение.
2. `docs/product/prd.md` — главный продуктовый источник истины.
3. Краткие operational docs (`mvp-boundaries.md`, `terminology.md`, `task-intake.md`).
4. Устные догадки ИИ и незафиксированные предположения.

## Рабочее правило для новых задач

Каждая feature-задача должна опираться хотя бы на один раздел `docs/product/prd.md` или `docs/product/decision-log.md`. Если явной опоры нет, задача считается потенциально неоднозначной и требует уточнения до реализации.
