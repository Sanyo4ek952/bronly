# Bronly

Проект использует документацию как обязательный источник истины для продуктовых, UX и технических решений.

## Где лежит истина

### Продукт

- `docs/product/prd.md` — основной продуктовый документ, canonical PRD, `source of truth`.
- `docs/product/decision-log.md` — зафиксированные решения и будущие продуктовые уточнения.
- `docs/product/mvp-boundaries.md` — краткие границы MVP и признаки scope creep.
- `docs/product/terminology.md` — словарь UI/UX терминов и запрещенных формулировок.

### Техническая реализация

- `docs/technical/architecture.md` — технический источник истины: стек, server/client границы, структура кода, данные, RLS, безопасность, PWA, производительность, проверки и деплой.

### Процесс

- `docs/process/task-intake.md` — чеклист для любой новой задачи.
- `docs/process/ai-acceptance-prompts.md` — контрольные промпты для быстрой проверки поведения ИИ.

## Как ИИ должен работать в этом проекте

Перед началом задачи ИИ должен прочитать:

1. этот `README.md`;
2. `docs/product/mvp-boundaries.md`;
3. `docs/product/terminology.md`;
4. при продуктовых или UX-изменениях — `docs/product/prd.md`;
5. при спорных или новых продуктовых решениях — `docs/product/decision-log.md`;
6. при изменении кода, архитектуры, данных, авторизации, RLS, PWA, уведомлений, зависимостей или деплоя — `docs/technical/architecture.md`.

Все текстовые файлы в репозитории должны сохраняться в `UTF-8 without BOM`. Для проверки кодировки и типичных следов mojibake используйте `npm run check:encoding`.

Если задача конфликтует с MVP, терминологией, технической архитектурой или зафиксированной бизнес-логикой, ИИ не должен молча реализовывать изменение. Вместо этого он обязан:

1. назвать конфликт;
2. сослаться на конкретный документ или раздел;
3. предложить 1-2 безопасных варианта;
4. дождаться решения пользователя.

## Приоритет документов

1. `docs/product/decision-log.md` — если там явно зафиксировано более новое продуктовое решение.
2. `docs/product/prd.md` — главный продуктовый источник истины.
3. `docs/technical/architecture.md` — главный источник истины по способу технической реализации.
4. Краткие operational docs (`mvp-boundaries.md`, `terminology.md`, `task-intake.md`).
5. Устные догадки ИИ и незафиксированные предположения.

Продуктовый документ отвечает на вопрос «что и зачем строим». Техническая архитектура отвечает на вопрос «как это безопасно реализуем». Техническое решение не может переопределять продуктовую модель.

## Рабочее правило для новых задач

Каждая feature-задача должна опираться хотя бы на один раздел `docs/product/prd.md` или `docs/product/decision-log.md` и не противоречить `docs/technical/architecture.md`. Если явной опоры нет, задача считается потенциально неоднозначной и требует уточнения до реализации.
