# Decision Log

Этот журнал хранит продуктовые решения, на которые ИИ и разработка должны опираться в первую очередь, если они новее кратких выжимок или уточняют спорные места.

## Статус документа

- Назначение: журнал зафиксированных решений и будущих уточнений.
- Приоритет: выше кратких operational docs, но ниже `prd.md`, если в журнале нет явного нового решения.
- Процесс: новые спорные решения сначала фиксируются здесь, а после стабилизации переносятся в `docs/product/prd.md`.

## Зафиксированные решения MVP

### Источник истины

- Canonical product spec: `docs/product/prd.md`.
- Краткие docs нужны как ускорители для ИИ, а не как замена полному ТЗ.

### Терминология

- В UI использовать язык заявок, а не язык гарантированного бронирования.

### Роли и права

- Владелец остается владельцем объекта.
- Агент может показывать объект в своей витрине только при активном сотрудничестве.
- Агент не редактирует чужой объект.
- Агент видит календарь занятости подключенного объекта только для чтения.

### Заявки и публичные страницы

- Заявка всегда создается только на конкретный `room_id`.
- Гость по ссылке агента видит контакты агента, а не владельца.
- Агентская заявка попадает владельцу только после действия агента “Передать владельцу”.

### Цены

- Ссылка показывает актуальную цену на момент открытия.
- Заявка сохраняет ценовой snapshot на момент отправки.
- Итоговая сумма считается по ночам, включая сезонные цены.
- В агентской витрине гость видит итоговую цену агента.

### Коллекции

- Коллекция может содержать объекты и/или отдельные номера.
- Даже если в коллекцию добавлен объект целиком, гость должен выбрать конкретный номер перед отправкой заявки.
- Для MVP рекомендовано: коллекция живет бессрочно, пока ее не архивировали.
- Для MVP рекомендовано: показывать базовую статистику открытия коллекции.
- Для MVP рекомендовано: коллекцию можно назвать под конкретного гостя.
- Для MVP рекомендовано: цены в коллекции показывать всегда.

### Агентский контур

- Статус `completed` ставит только владелец объекта.
- Агент может запросить у владельца отметить сделку как `completed`.
- Статус “спор” в MVP не вводится.

## 2026-06-05 - Автоматическая публичная ссылка агента

- Контекст: ручной `slug` агента приводил к нерабочим публичным ссылкам, особенно при кириллице. Для MVP нужно убрать ручной ввод и сохранить стабильный URL агентской витрины.
- Решение: публичная ссылка агента генерируется автоматически и использует отдельный стабильный идентификатор формата `/a/ag_xxxxxx`. Пользователь не редактирует его вручную. Старые агентские ссылки на основе `slug` должны редиректить на новый канонический URL.
- Что меняется: в агентском кабинете показывается только готовая публичная ссылка; `slug` больше не используется как канонический URL агента; legacy `slug` остается техническим источником для совместимости и редиректа.
- Нужно ли перенести в prd.md: да, при следующем обновлении раздела про агентскую витрину и публичные страницы.

## 2026-06-07 - Самостоятельные номера без объекта

- Контекст: часть владельцев работает не с объектами из нескольких номеров, а с одним самостоятельным вариантом размещения. Модель только `object -> rooms` создавала лишний искусственный контейнер и мешала единому списку в кабинете.
- Решение: владелец может создать либо объект, либо отдельный номер. Отдельный номер существует без привязки к объекту, имеет собственные адресные поля и тип размещения, попадает в общий owner-список и в отдельный публичный блок владельца.
- Что меняется: `room` перестает быть только дочерней сущностью объекта; публичная owner-страница показывает и объекты, и отдельные номера; заявка по-прежнему всегда создается только на конкретный номер; агентский контур должен поддерживать и отдельные номера тоже.
- Нужно ли перенести в prd.md: да.
