# Границы MVP Bronly

Этот документ — короткая operational-выжимка из `docs/product/prd.md`. При расхождении приоритет у `prd.md`, если `decision-log.md` не фиксирует более новое решение.

## Что входит в MVP

- mobile-first PWA/SaaS для владельцев жилья и агентов;
- персональные публичные страницы владельца и агента;
- объекты, номера, фото, цены и календарь занятости;
- ручное управление занятыми датами;
- заявки от гостей на конкретный номер;
- owner/agent логика источника заявки;
- коллекции объектов и номеров;
- подписка с ограничением доступа;
- Telegram, in-app и PWA push уведомления;
- базовые юридические чекбоксы и тексты ответственности.

## Что точно не входит без отдельного решения

- общий каталог объектов;
- глобальный поиск по всем владельцам;
- рекомендации похожих объектов;
- рейтинги и отзывы;
- онлайн-оплата проживания;
- автоматическое подтверждение заявки;
- гарантированное проживание;
- возвраты денег гостям;
- интеграции с Booking, Авито, Суточно, Островок и channel manager;
- App Store / Google Play приложения;
- SMS-уведомления;
- сложный CRM;
- роли персонала владельца;
- автодоговор между владельцем и агентом;
- автоматические взаиморасчеты между владельцем и агентом;
- публичный каталог агентов для гостей;
- мультиязычность на первом этапе без отдельного решения.

## Ключевые продуктовые ограничения

- Bronly не является агрегатором или маркетплейсом.
- Сервис не подтверждает проживание и не принимает оплату за проживание.
- Гость видит только контент в рамках конкретной ссылки владельца, агента или коллекции.
- Общего каталога для гостей нет.
- Одна заявка всегда создается только на конкретный номер.
- Агент не редактирует чужой объект.

## Когда ИИ должен считать запрос scope creep

Запрос считается выходом за рамки текущего MVP, если он:

- превращает сервис в каталог или маркетплейс;
- добавляет платежную логику по проживанию;
- обещает гарантированное бронирование от имени сервиса;
- ломает зафиксированную модель ролей owner/agent/guest;
- расширяет права агента на редактирование чужого объекта;
- меняет смысл терминов в UI на бронировочную лексику.

## Что делать при конфликте

ИИ не должен внедрять конфликтующее решение молча. Нужно:

1. назвать конфликт;
2. указать документ;
3. предложить 1-2 безопасных варианта;
4. дождаться решения пользователя.
