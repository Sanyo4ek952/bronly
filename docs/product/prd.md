# ТЗ v1.20 Final Clean на MVP: Bronly — SaaS-сервис персональных страниц для владельцев жилья

## 0. Краткое описание проекта

Bronly — mobile-first PWA/SaaS-сервис для владельцев жилья, гостевых домов, мини-отелей, комнат, номеров и агентов, которые помогают владельцам получать заявки.

Владелец создает свои объекты, номера, фото, цены и календарь занятости. После этого он получает персональную публичную страницу, которую может отправлять гостям.

Агент может создать свою агентскую витрину и показывать в ней собственные объекты или объекты владельцев, с которыми есть активное сотрудничество.

Гость открывает конкретную ссылку владельца, агента или коллекции и видит только варианты в рамках этой ссылки. Bronly не является агрегатором, маркетплейсом или сервисом гарантированного бронирования.

## 1. Цели MVP

- Дать владельцу простой mobile-first кабинет для ведения объектов, номеров, цен, фото и занятых дат.
- Дать владельцу публичную персональную страницу для отправки гостям.
- Дать агенту отдельную витрину для работы с объектами владельцев при активном сотрудничестве.
- Позволить гостю оставить заявку на конкретный номер.
- Зафиксировать источник заявки: владелец, агент или коллекция.
- Поддержать подписочную модель доступа.
- Поддержать Telegram, in-app и PWA push уведомления.
- Сохранить юридическую рамку: сервис не гарантирует проживание, не принимает оплату за проживание и не подтверждает заявку от имени владельца.

## 2. Что не входит в MVP

Без отдельного продуктового решения не реализовывать:

- общий каталог объектов;
- глобальный поиск по всем владельцам;
- рекомендации похожих объектов;
- рейтинги и отзывы;
- онлайн-оплату проживания;
- автоматическое подтверждение заявки;
- гарантированное проживание;
- возвраты денег гостям;
- интеграции с Booking, Авито, Суточно, Островок, channel manager и другими площадками;
- App Store / Google Play приложения;
- SMS-уведомления;
- сложный CRM;
- роли персонала владельца;
- автоматический договор между владельцем и агентом;
- автоматические взаиморасчеты между владельцем и агентом;
- публичный каталог агентов для гостей;
- мультиязычность на первом этапе.

## 3. Терминология

### Использовать в UI

- заявка;
- запрос на проживание;
- календарь занятости;
- занятые даты;
- свободные даты;
- отправить заявку;
- оставить запрос;
- принята владельцем;
- владелец свяжется с вами;
- уточнить доступность.

### Не использовать в UI

- гарантированная бронь;
- забронировано сервисом;
- подтверждение от сервиса;
- оплатить бронь;
- мгновенное бронирование;
- сервис подтвердил бронь.

В коде допустимы технические сущности со словом `booking`, если это упрощает реализацию. В пользовательских текстах должны использоваться формулировки `заявка` и `запрос на проживание`.

## 4. Роли пользователей

### 4.1. Владелец

Владелец управляет своими объектами, номерами, фото, ценами, занятыми датами, публичной страницей, заявками и подпиской.

Владелец может:

- создавать и редактировать свои объекты;
- создавать и редактировать отдельные номера без объекта;
- создавать и редактировать номера;
- загружать фото объектов и номеров;
- задавать базовые и сезонные цены;
- отмечать занятые даты;
- принимать заявки от гостей;
- видеть источник заявки;
- управлять сотрудничеством с агентами;
- включать или выключать показ объекта агенту;
- создавать коллекции объектов и номеров;
- получать уведомления;
- оплачивать или продлевать подписку вручную в рамках MVP.

### 4.2. Агент

Агент создает агентскую витрину и работает с заявками, которые пришли по его ссылкам.

Агент может:

- настроить профиль и публичную агентскую страницу;
- показывать свои объекты;
- отправлять владельцам предложения о сотрудничестве;
- показывать объекты владельцев только при активном сотрудничестве;
- видеть календарь занятости подключенного объекта только для чтения;
- создавать коллекции;
- принимать агентские заявки;
- передавать заявку владельцу;
- запросить у владельца отметить сделку как `completed`.

Агент не редактирует чужой объект, номер, фото, календарь владельца или базовые цены владельца.

### 4.3. Гость

Гость открывает конкретную публичную ссылку владельца, агента или коллекции. Общего каталога для гостя нет.

Гость может:

- посмотреть доступные объекты и номера в рамках конкретной ссылки;
- выбрать даты, количество гостей и комнат;
- увидеть цену за сутки и итоговую сумму;
- оставить заявку только на конкретный номер;
- передать имя, телефон и комментарий.

Гость не получает гарантированного подтверждения от сервиса. После отправки заявки владелец или агент связывается с гостем вне Bronly.

### 4.4. Администратор сервиса

Администратор — владелец платформы.

Администратор может:

- просматривать список пользователей;
- просматривать список объектов;
- смотреть статус подписок;
- вручную продлевать подписку;
- скрывать публичные страницы при нарушениях;
- блокировать объект при нарушениях.

## 5. Основные продуктовые сценарии

### 5.1. Вход и регистрация

Основные маршруты:

```txt
/login
/register
/check-email
```

При регистрации пользователь выбирает роль. В MVP рекомендуется поддержать техническую возможность нескольких ролей через таблицу `user_roles`.

### 5.2. Онбординг владельца

```txt
/dashboard
/dashboard/properties
/dashboard/properties/new
/dashboard/properties/[id]
/dashboard/calendar
/dashboard/requests
/dashboard/subscription
```

Владелец должен быстро пройти путь:

1. создать профиль;
2. создать объект;
3. добавить номер или создать отдельный номер;
4. добавить фото;
5. задать цену;
6. отметить занятые даты;
7. скопировать публичную ссылку.

### 5.3. Публичная страница владельца

Публичная страница владельца показывает только его объекты и номера.
Отдельные номера без объекта показываются отдельным блоком на той же странице владельца.

Требования:

- mobile-first;
- нет общего каталога;
- нет поиска по другим владельцам;
- есть карточки объектов и номеров;
- отдельные номера без объекта показываются отдельным блоком;
- есть фильтр по датам, гостям и комнатам;
- неподходящие варианты остаются видимыми ниже с понятной пометкой;
- заявка создается только на конкретный `room_id`;
- контакты владельца показываются в рамках его страницы.

### 5.4. Агентская витрина

Агентская витрина показывает:

- собственные объекты агента;
- объекты владельцев, с которыми есть активное сотрудничество;
- контакты агента, а не владельца.

Если гость открыл ссылку агента, заявка считается агентской заявкой. Агентская заявка попадает владельцу только после действия агента “Передать владельцу”.

### 5.5. Сотрудничество владельца и агента

Базовый сценарий:

1. Владелец по желанию открывает объект для предложений агентов.
2. Агент видит объект в контуре доступных для предложений объектов.
3. Агент отправляет предложение владельцу.
4. Владелец принимает или отклоняет предложение.
5. После принятия создается активная связь агента и объекта.
6. Объект появляется у агента в агентской витрине.
7. Агент может принимать заявки по своей ссылке.
8. Агент может передать заявку владельцу.
9. Статус `completed` ставит только владелец объекта.

Ограничения:

- Bronly не рассчитывает комиссию агента в MVP;
- Bronly не выплачивает вознаграждение агенту;
- Bronly не является стороной договора между владельцем и агентом;
- спорный статус по агентской сделке в MVP не вводится.

### 5.6. Коллекции

Коллекция — подборка объектов и/или отдельных номеров, которую владелец или агент отправляет гостю отдельной ссылкой.

Требования MVP:

- коллекция может содержать объекты и/или номера;
- если добавлен объект целиком, гость все равно выбирает конкретный номер перед отправкой заявки;
- коллекция живет бессрочно, пока ее не архивировали;
- коллекцию можно назвать под конкретного гостя;
- цены в коллекции показываются всегда;
- показывается базовая статистика открытия коллекции;
- заявка создается только на конкретный `room_id`.

## 6. Заявки

Одна заявка всегда создается только на конкретный номер.
Если номер самостоятельный и не привязан к объекту, заявка все равно создается на `room_id`, а `property_id` может отсутствовать.

Заявка должна хранить:

- источник: `owner`, `agent`, `collection`;
- `property_id`;
- `room_id`;
- `owner_id`;
- `agent_id`, если применимо;
- данные гостя;
- даты проживания;
- количество гостей;
- количество комнат;
- ценовой snapshot на момент отправки;
- статус заявки.

Статусы MVP:

```txt
new
accepted_by_owner
rejected
transferred_to_owner
completed
```

Публичный текст после отправки заявки не должен обещать подтверждение от сервиса. Безопасная формулировка: “Заявка отправлена. Владелец свяжется с вами для уточнения доступности.”

## 7. Цены

### 7.1. Базовые правила

- Владелец задает базовую цену номера.
- Владелец может задать сезонные цены.
- В сезонной цене не указывается название сезона.
- Для одного номера нельзя создать пересекающиеся активные сезонные цены.
- Ссылка показывает актуальную цену на момент открытия.
- Заявка сохраняет цену на момент отправки.
- Итоговая сумма считается по ночам.
- В агентской витрине гость видит итоговую цену агента.

### 7.2. Расчет итоговой суммы

```txt
check_in = дата заезда
check_out = дата выезда
nights = check_out - check_in
total = сумма цены каждой ночи
```

Если на ночь действует сезонная цена, используется сезонная цена. Иначе используется базовая цена номера.

### 7.3. Агентские цены

Агент может показывать гостю итоговую цену агента в рамках своей витрины. При этом Bronly не выполняет автоматические взаиморасчеты между владельцем и агентом.

## 8. Подписка и оплата

Подписка считается по количеству активных номеров во всех объектах пользователя.

MVP допускает ручную оплату:

```txt
Пользователь оплачивает вне автоматического платежного контура.
Администратор вручную продлевает доступ.
```

Grace period:

```txt
active -> grace_period -> expired
```

В grace period:

- публичные страницы могут оставаться доступными;
- пользователь видит предупреждение;
- администратор может вручную продлить доступ.

После окончания grace period:

- статус подписки становится `expired`;
- публичная страница может быть скрыта;
- новые заявки могут быть ограничены.

Онлайн-оплата проживания не входит в MVP.

## 9. Mobile-first и PWA

Весь интерфейс проектируется сначала под телефон.

Приоритетные устройства:

- Android-смартфоны;
- iPhone;
- планшеты;
- desktop как вторичная адаптация.

Приложение должно работать как PWA:

- установка на главный экран;
- корректные safe-area отступы;
- PWA push-уведомления;
- Telegram-уведомления;
- in-app уведомления.

Мобильная навигация владельца:

```txt
Главная | Номера | Календарь | Заявки | Ещё
```

## 10. Уведомления

В MVP поддерживаются:

- Telegram;
- in-app уведомления;
- PWA push.

SMS-уведомления не входят в MVP.

События для уведомлений:

- новая заявка;
- заявка передана владельцу;
- предложение агента владельцу;
- принятие или отклонение предложения;
- напоминание о подписке;
- изменение статуса подписки.

## 11. Юридические ограничения

Bronly должен явно фиксировать:

- сервис не является стороной договора проживания;
- сервис не подтверждает проживание;
- сервис не гарантирует доступность номера;
- сервис не принимает оплату за проживание;
- владелец или агент самостоятельно связывается с гостем;
- пользователь соглашается с обработкой персональных данных.

- номера владельца;

- календарь;
- заявки;

Правило по умолчанию: пользователь видит и меняет только свои данные. Агент видит объект владельца только в объеме, необходимом для активного сотрудничества, и не редактирует чужие данные.

Backend-only операции выполняются через service role и не должны быть доступны из клиентского кода.

## 17. Критерии готовности MVP

MVP считается готовым, если:

1. Владелец может создать объект и номер либо отдельный номер без объекта.
2. Владелец может добавить фото и цены.
3. Владелец может отметить занятые даты.
4. Гость может открыть публичную ссылку владельца.
5. Гость может оставить заявку на конкретный номер.
6. Заявка сохраняет источник и ценовой snapshot.
7. Агент может настроить витрину.
8. Агент может показывать объект владельца только при активном сотрудничестве.
9. Агент не редактирует чужой объект.
10. Агентская заявка передается владельцу только действием агента.
11. Владелец ставит `completed`.
12. Коллекция может содержать объекты и номера.
13. В коллекции заявка создается только на конкретный номер.
14. Итоговая сумма считается по ночам с учетом сезонных цен.
15. Подписка ограничивает доступ по активным номерам.
16. Уведомления работают в Telegram, in-app и PWA push.
17. UI не использует запрещенную бронировочную лексику.

## 20. Открытые вопросы по агентской модели, коллекциям и ценам

Зафиксировано:

- коллекция может содержать объекты и/или номера;
- при объекте в коллекции гость выбирает конкретный номер;
- ссылка показывает актуальную цену на момент открытия;
- заявка сохраняет цену на момент отправки;
- статус `completed` ставит владелец;
- спорный статус в MVP не вводится;
- Bronly не делает автоматические взаиморасчеты между владельцем и агентом.

Рекомендовано для MVP:

- коллекция живет бессрочно, пока ее не архивировали;
- показывать базовую статистику открытия коллекции;
- разрешить название коллекции под конкретного гостя;
- цены в коллекции показывать всегда.

## 21. Резюме

Bronly в MVP — это не маркетплейс и не сервис гарантированного бронирования. Это персональная витрина владельца и агента, которая помогает быстро показать гостю конкретные варианты и получить заявку.

Ключевая модель:

- владелец управляет своими объектами;
- агент показывает объект только при активном сотрудничестве;
- гость оставляет заявку только на конкретный номер;
- Bronly не подтверждает проживание и не принимает оплату за проживание;
- подписка дает доступ к сервису;
- UI говорит языком заявок, а не гарантированной брони.
