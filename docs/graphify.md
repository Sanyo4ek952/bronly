# Graphify в Bronly

Graphify в Bronly нужен как навигатор по архитектуре проекта. Он помогает быстрее понять связи между страницами, компонентами, shared-логикой и соседними модулями, но не заменяет чтение кода и не является постоянной памятью сам по себе.

## Зачем он нужен

Используем Graphify, чтобы:
- понять архитектуру проекта перед крупной задачей;
- найти связи между модулями и слоями;
- подготовить безопасный рефакторинг;
- быстро сузить круг файлов для чтения;
- разобраться, какие части кода участвуют в конкретном flow.

## Два режима работы

### Тяжёлый режим

Используем для средних и крупных задач:
- архитектурные изменения;
- cross-module задачи;
- refactoring;
- onboarding в незнакомую область;
- поиск неочевидных связей между файлами.

В этом режиме агент должен:
1. выбрать минимальный scope;
2. собрать или обновить граф по этому scope;
3. прочитать только релевантные файлы;
4. внести изменения;
5. после принятия обновить граф.

### Лёгкий режим

Используем для локальных задач:
- поправить текст;
- изменить цвет или padding;
- заменить одну кнопку;
- исправить локальную ошибку в одном понятном компоненте.

В этом режиме Graphify можно не запускать, если область задачи и так очевидна.

## Рекомендованный workflow

Для задач, где Graphify нужен:
1. Получить задачу и оценить её scope.
2. Выбрать минимальную область анализа:
   - `src`
   - `src/app`
   - `src/components`
   - `src/shared`
3. Собрать или обновить граф по этой области.
4. Использовать граф как навигатор и только потом читать нужные файлы.
5. Выполнить изменения.
6. После принятия изменений обновить граф через `graphify update`.

Graphify не нужно запускать на весь репозиторий без явной причины. Для Bronly обычно достаточно `src` или более узкой подпапки.

## Команды

Проверка CLI:

```bash
npm run graphify:check
```

Основной запуск по исходному коду:

```bash
npm run graphify:src
```

Точечный запуск:

```bash
npm run graphify:app
npm run graphify:components
npm run graphify:shared
```

Дешёвое обновление графа после принятых изменений:

```bash
graphify update ./src
```

Если задача была локальной, можно обновлять более узкий путь, например:

```bash
graphify update ./src/components
```

## Когда использовать

Graphify нужен, если задача:
- затрагивает несколько модулей;
- требует понять flow данных или пользовательский сценарий;
- связана с архитектурой;
- затрагивает неочевидные зависимости;
- может привести к рефакторингу.

## Когда не использовать

Graphify не нужен, если задача:
- ограничена одним понятным файлом;
- меняет только текст;
- меняет только стили;
- исправляет маленькую локальную ошибку;
- не требует разбираться в соседних модулях.

## Что делать, если квота или API недоступны

Если semantic extraction не работает из-за квоты, сети или API:
- не блокируйте задачу;
- используйте AST-only сценарий через `graphify update ./src`;
- продолжайте анализ по локальному графу и обычному чтению кода.

Это fallback, а не ошибка процесса.

## Почему нельзя без необходимости запускать Graphify на весь проект

Полный запуск по репозиторию может захватить лишние данные:
- сгенерированные файлы;
- public-assets;
- медиафайлы;
- служебные директории;
- env-файлы и lock-файлы.

Это увеличивает шум, замедляет анализ и может тратить лишние токены без пользы для архитектурной задачи.

## Игнорируемые директории и файлы

Graphify не должен анализировать:
- `node_modules`
- `.next`
- `dist`
- `build`
- `out`
- `coverage`
- `.git`
- `.turbo`
- `.vercel`
- `supabase/.temp`
- `public`
- env-файлы
- lock-файлы
- IDE-папки
- логи
- изображения, видео, pdf и архивы
