# Техническая архитектура Bronly

## Статус документа

- Назначение: технический источник истины для разработки Bronly.
- Область: стек, границы клиентской и серверной частей, структура кода, данные, безопасность, PWA, уведомления, качество и деплой.
- Приоритет: продуктовые решения определяются `docs/product/prd.md` и `docs/product/decision-log.md`; этот документ определяет способ их технической реализации.
- При конфликте с фактическим кодом нельзя молча закреплять расхождение: нужно либо привести код к документу, либо зафиксировать новое техническое решение.

## 1. Архитектурная цель

Bronly строится как mobile-first PWA/SaaS на Next.js. Приложение включает:

1. публичные страницы владельца, агента и коллекций;
2. защищенный кабинет владельца;
3. защищенный кабинет агента;
4. административный контур;
5. серверную работу с пользователями, объектами, номерами, календарем, заявками, подписками и уведомлениями;
6. PostgreSQL с обязательной защитой данных через RLS.

Bronly не является агрегатором, маркетплейсом или сервисом гарантированного бронирования. Технические решения не должны незаметно превращать продукт в одну из этих моделей.

## 2. Основной стек

### 2.1. Приложение

- Next.js 16, App Router;
- React 19;
- TypeScript в строгом режиме;
- Server Components по умолчанию;
- Client Components только для интерактивности, браузерных API и локального UI-состояния;
- `next/image`, Metadata API и серверные возможности Next.js для публичных страниц и SEO.

### 2.2. Данные и авторизация

- Supabase Auth;
- PostgreSQL в Supabase;
- Supabase Storage для изображений;
- Row Level Security как обязательный уровень доступа;
- `@supabase/ssr` для корректной работы сессии на сервере и в браузере;
- SQL-миграции как единственный штатный способ изменения схемы удаленной БД.

### 2.3. Интерфейс

- собственная система CSS design tokens;
- переиспользуемые компоненты в `src/shared/ui`;
- Lucide React для иконок;
- mobile-first верстка;
- поддержка safe-area для установленных PWA;
- единый визуальный язык публичного сайта, кабинетов и админки.

Не добавлять UI-фреймворк, глобальный state manager или form-библиотеку только ради одной задачи. Новая зависимость должна решать повторяющуюся проблему и быть обоснована.

### 2.4. PWA и уведомления

- Web App Manifest;
- service worker;
- установка приложения на главный экран;
- in-app уведомления;
- Telegram Bot API;
- Web Push после готовности базовых продуктовых сценариев.

### 2.5. Размещение

- Vercel для Next.js;
- Supabase для Auth, PostgreSQL и Storage;
- секреты хранятся только в переменных окружения платформы;
- service role используется только в server-only коде.

## 3. Принцип разделения ответственности

### `src/app`

Отвечает за:

- маршруты;
- layouts;
- loading/error/not-found состояния;
- серверную композицию страниц;
- route handlers;
- metadata и SEO;
- границы публичного и защищенного контуров.

Страницы не должны превращаться в большие файлы с бизнес-логикой. Они собирают сценарий из компонентов и серверных функций.

### `src/features`

Отвечает за законченные пользовательские действия и сценарии, например:

- регистрация и вход;
- создание объекта;
- создание отдельного номера;
- изменение занятых дат;
- отправка заявки;
- передача агентской заявки владельцу;
- управление подпиской;
- регистрация service worker.

Фича не должна напрямую менять данные другой фичи через внутренние детали. Общая логика выносится ниже.

### `src/shared`

Отвечает за:

- UI-примитивы;
- design tokens и общие стили;
- Supabase clients;
- конфигурацию;
- общие типы;
- форматирование;
- работу с датами и ценами;
- SEO helpers;
- общие server-only utilities.

В `shared` нельзя помещать бизнес-правила конкретной роли или страницы только ради удобного импорта.

### `src/components`

Допускается для составных визуальных блоков, которые собирают несколько UI-примитивов, но сами не владеют бизнес-правилами и доступом к данным.

### Развитие структуры

Архитектура FSD-inspired, но без создания пустых слоев. Новые `entities`, `widgets` или дополнительные уровни вводятся только тогда, когда код реально повторяется и существующая структура перестает отражать ответственность.

## 4. Серверный и клиентский код

### Сервер по умолчанию

На сервере выполняются:

- чтение защищенных данных;
- проверка авторизации и ролей;
- создание и изменение сущностей;
- расчет итоговой цены;
- формирование ценового snapshot;
- проверка подписки и лимитов;
- административные действия;
- операции с service role;
- отправка уведомлений.

### Клиент только при необходимости

Client Component используется, когда нужен:

- обработчик пользовательского ввода;
- локальное раскрытие/переключение;
- drag-and-drop;
- browser API;
- service worker;
- push subscription;
- оптимистическое UI-поведение.

Нельзя переносить на клиент секреты, авторизационные проверки или правила доступа ради упрощения интерфейса.

## 5. Доступ к данным

### Обязательный поток изменения данных

1. Пользователь отправляет действие.
2. Сервер проверяет сессию.
3. Сервер проверяет роль и право на конкретную сущность.
4. Входные данные проходят runtime-валидацию.
5. PostgreSQL/RLS повторно ограничивает доступ.
6. Изменение выполняется транзакционно, если затронуто несколько связанных сущностей.
7. Сервер возвращает безопасный результат без секретных полей.
8. Связанный кэш и UI обновляются целевым способом.

TypeScript-типы не заменяют runtime-валидацию данных, полученных из формы, URL, webhook или внешнего API.

### Публичные страницы

Публичное чтение разрешает только данные, которые владелец или агент явно опубликовал и которые доступны при текущем статусе подписки и блокировки.

Публичный запрос не должен возвращать:

- приватные контакты, не предназначенные для витрины;
- внутренние заметки;
- данные других владельцев;
- служебные поля подписки;
- service-role данные;
- персональные данные гостей и заявки.

## 6. Обязательные инварианты модели данных

1. Заявка всегда создается на конкретный `room_id`.
2. Отдельный номер может существовать без `property_id`.
3. Если заявка относится к отдельному номеру, `property_id` может быть `null`.
4. Заявка сохраняет источник: owner, agent или collection.
5. Заявка сохраняет ценовой snapshot на момент отправки.
6. Итоговая сумма считается по ночам, с учетом сезонной цены каждой ночи.
7. Активные сезонные цены одного номера не пересекаются.
8. Агент не редактирует объект, номер, фото, календарь или базовую цену владельца.
9. Агент видит чужой объект только в рамках активного сотрудничества.
10. Статус `completed` агентской сделки ставит только владелец.
11. Публичная страница не обещает подтверждение проживания от имени Bronly.
12. Истечение подписки влияет на доступ и публикацию, но не должно удалять данные пользователя.

Эти правила должны быть защищены не только интерфейсом, но и серверной логикой, ограничениями БД или RLS там, где это возможно.

## 7. Авторизация и роли

Поддерживаемые роли:

- `owner`;
- `agent`;
- `admin`;
- `guest` не требует регистрации для публичной заявки.

Проверка доступа состоит из двух уровней:

1. аутентификация пользователя;
2. авторизация на конкретное действие и конкретную запись.

Наличие роли `owner` не дает права редактировать любой объект. Необходимо проверять владение записью. Наличие роли `agent` не дает права читать любой календарь: нужна активная связь с объектом.

Административный интерфейс не должен полагаться только на скрытую ссылку или клиентский redirect. Право администратора проверяется на сервере.

## 8. RLS и безопасность

Для каждой пользовательской таблицы до использования в production должны существовать:

- включенный RLS;
- явные политики `select`, `insert`, `update`, `delete`;
- проверка владельца или допустимой связи;
- отдельное решение для публичного чтения;
- тест негативного сценария доступа.

Правила:

- service role никогда не импортируется в Client Component;
- секретные env-переменные не имеют префикс `NEXT_PUBLIC_`;
- пользователь не передает доверенный `owner_id` или роль без серверной проверки;
- пути Storage и политики должны связывать файл с владельцем сущности;
- загрузка файлов ограничивается по MIME type, размеру и количеству;
- ошибки не возвращают SQL, токены и внутренние детали;
- rate limit обязателен для публичной отправки заявок, входа и чувствительных действий;
- webhook проверяет подпись или секрет источника;
- все операции администратора должны оставлять понятный след в данных или логах.

## 9. Работа с датами и ценами

- Даты проживания хранятся как календарные даты, а не как локальное время браузера.
- Интервал проживания трактуется как `[check_in, check_out)`: день выезда не является оплачиваемой ночью.
- Расчет цены выполняется на сервере.
- Клиентский расчет может использоваться только как предварительный UI-preview.
- Денежные значения хранятся целым числом в минимальной денежной единице либо в другом заранее зафиксированном точном формате; `float` для денег не используется.
- Валюта должна быть явным полем или системным инвариантом, а не догадкой интерфейса.
- После отправки заявки изменение текущих цен не меняет сохраненный snapshot заявки.

## 10. Изображения и Storage

- Оригинал и отображаемые изображения не должны загружаться без ограничений.
- В публичной выдаче используется оптимизированное изображение и корректный `sizes`.
- Для объекта и номера хранится управляемый порядок фотографий.
- Удаление сущности должно учитывать связанные файлы и не оставлять бесконечно растущие orphan-файлы.
- Публичный URL не должен давать возможность заменить или удалить чужое изображение.

## 11. Производительность

- Не загружать все объекты, номера, заявки или события календаря без пагинации или диапазона дат.
- Календарь запрашивает только нужный период.
- Списки в кабинетах используют серверную пагинацию или cursor-based загрузку.
- Публичная страница оптимизирована для первого мобильного открытия по ссылке.
- Тяжелые интерактивные блоки загружаются только там, где они нужны.
- Новая зависимость оценивается по влиянию на клиентский bundle.
- Кэш публичных данных не должен показывать устаревшую занятость после изменения календаря дольше явно принятого продуктового интервала.

## 12. PWA

PWA не должна маскировать устаревшие данные.

Разрешено кэшировать:

- статические ресурсы;
- шрифты;
- иконки;
- shell приложения;
- безопасные публичные ответы с понятной стратегией обновления.

Нельзя без отдельного решения кэшировать как источник истины:

- заявки;
- подписки;
- права пользователя;
- актуальную занятость;
- административные данные.

При недоступной сети интерфейс должен честно показывать, какие данные не удалось обновить, и не подтверждать изменение до ответа сервера.

## 13. Уведомления

Уведомление строится вокруг доменного события, а не напрямую внутри UI-компонента.

Рекомендуемый поток:

1. сервер успешно завершает бизнес-операцию;
2. создается запись доменного события или задания доставки;
3. обработчик формирует сообщения для разрешенных каналов;
4. результат доставки сохраняется;
5. повторная попытка не дублирует бизнес-действие.

Отказ Telegram или Push не должен откатывать успешно сохраненную заявку. Пользователь должен увидеть заявку в кабинете даже при сбое внешнего канала.

## 14. UI и design system

- Перед созданием компонента искать существующий аналог.
- Повторяющийся визуальный паттерн выносить в общий компонент.
- Цвета, радиусы, отступы, типографика, тени и состояния берутся из design tokens.
- Не создавать произвольные локальные цвета и размеры без причины.
- Состояния loading, empty, error, disabled и success проектируются вместе с основным экраном.
- Все ключевые сценарии проверяются на мобильной ширине.
- Интерактивные элементы имеют доступный размер, focus state и понятную подпись.
- Тексты следуют `docs/product/terminology.md`.

## 15. Качество и проверки

### Минимум для каждой задачи

- TypeScript не получает новых ошибок в затронутой области;
- ESLint не получает новых ошибок в затронутых файлах;
- измененный сценарий проверен вручную;
- критичное бизнес-правило покрыто тестом, когда его можно надежно проверить автоматически;
- SQL-изменение имеет миграцию и проверку прав доступа.

### Экономная стратегия проверок

Для небольшой локальной правки сначала выполняются только связанные проверки:

- тест измененного модуля;
- typecheck или сборка затронутой области, если доступно;
- lint затронутых файлов;
- ручной сценарий.

Полный lint/build и расширенные тесты обязательны, когда меняются:

- общая архитектура;
- зависимости;
- конфигурация Next.js, TypeScript, ESLint или PWA;
- auth и session flow;
- RLS или общие политики доступа;
- shared-компоненты с широким использованием;
- схема БД и критические миграции;
- расчет цен, подписки или источника заявки.

## 16. Деплой и окружения

Минимальные окружения:

- local;
- preview для ветки или pull request;
- production.

Правила:

- production-ключи не используются локально;
- миграции проверяются до production;
- preview не должен отправлять реальные уведомления без явного разрешения;
- внешние callback URL задаются отдельно для окружений;
- перед production-деплоем проверяются auth redirect URLs, RLS, Storage policies, env и публичные формы;
- изменение схемы должно быть совместимо с текущим приложением на время выкладки либо выпускаться в безопасной последовательности.

## 17. Definition of Done

Техническая задача считается завершенной, если:

1. реализация соответствует PRD или свежему decision log;
2. не нарушены продуктовые инварианты;
3. доступ проверяется на сервере и через RLS;
4. интерфейс работает mobile-first;
5. использованы существующие tokens и компоненты;
6. обработаны loading, empty и error состояния;
7. выполнены проверки, соответствующие области риска;
8. документация обновлена, если изменилось устойчивое правило, стек, схема или архитектура;
9. в код не попали секреты и персональные данные;
10. изменение можно безопасно развернуть и откатить.

## 18. Изменение архитектуры

Новое устойчивое техническое решение сначала фиксируется в этом документе или отдельном ADR.

ADR нужен, если изменение:

- добавляет основную инфраструктурную зависимость;
- меняет способ авторизации;
- меняет границу server/client;
- меняет модель доступа или RLS;
- вводит очередь, фоновые задания или внешний сервис;
- меняет стратегию PWA-кэширования;
- меняет формат хранения денег, дат или цен;
- затрагивает несколько основных контуров приложения.

Шаблон ADR:

```md
# ADR-XXXX: Название

- Статус: proposed | accepted | superseded
- Контекст:
- Решение:
- Альтернативы:
- Последствия:
- План миграции:
```
